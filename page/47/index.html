到底什么是数据模型和数据建模

一直比较困惑于在移动互联网中的数据模型设计到底应该是怎么样的？原先在做移动项目的时候，往往有较成熟的eTom模型或者BOSS规范，从初始开始设计时就会从概念模型的角度出发。但现在往往被领导、产品经理、数据分析师所提的名词所困惑住，什么数据模型、数据建模、数据埋点、数据体系、指标体系，很多时候都在混用，但到底每个人心目中的数据模型或者建模又是指什么呢？很多时候这里还貌似没有开发的什么事？我就更迷糊了。

最近和产品、数据的同事有了进一步的交流，把自己的一点体会再理理。不同的角色对于数据的出发点和需求不同，而在一个系统的整体设计和实现中，数据的体现有不同层次，而数据也必然是和产品和业务相结合的。完整的业务模型应该包括数据模型和业务流程。而在数据体系上，从层次上我们可以划分为：指标体系 - 概念模型 - 逻辑模型 - 物理模型。（仓库的底层模型设计虽然不同，但也归结为此列）

在产品设计的角度上往往看到的比较多的是UI界面，大部分PRD中提供的都是一个手机页面再接着一个手机页面，比如：

下载 -》访问 -》注册 -》登录 -》游戏

这个是业务流程，做得比较细的产品经理会在流程上标注出核心的业务数据。这也就是我们现阶段通常所说的数据埋点了。

而从领导的角度来说，他更多看得是产品的成功与否如何来做衡量？这样往往一个产品设计开始就会先制定一些KPI指标，比如用户活跃度。如果指标是多维度的来做衡量的，这就是指标体系。（一个好的指标体系肯定不会是以单个指标来衡量的，比如谷歌衡量用户体验的指标体系提到的HEART）

但是我认为这些都还不是数据建模或者数据模型，数据模型通常是会从中抽象出来概念模型，再经过设计和开发人员逐步细化到逻辑和物理模型。基于业务流程的数据埋点往往会偏重于主流程，分支可能会被忽略，另外维度也很可能考虑不全；而基于指标体系的数据设计关注点也会偏于狭窄，并且指标只是计算的结果，而如何计算得到的才是关键。而要保证从产品设计到最终实现，数据指标定义和提取的一致性，模型的建立和一致的理解还是非常关键的。

我拿网上的一个Facebook设计主管谈如何利用数据做决策来简单分析一下之间的区别：

Julie讲述了自己的产品设计团队通过数据来分析用户是如何利用不同功能的，她以图片上传功能为例进行讲述。她的团队得出了以下几个数据：

  • 87%的用户在第一屏中的相册专辑名字提示框中选择类型
  • 57%的用户打开文件选择功能来选择他们想上传的图片
  • 52%的用户点击上传按钮
  • 48%的用户会等待上传进度完成

问题：少于50%的用户能够成功上传图片

解决方案：为了提高上传成功率，Facebook将Java/flash facebook文件选择功能改成浏览器原生文件选择功能，结果上传量提高11%。为了提高上传效率，在上传前不显示专辑创建功能。

问题：团队发现上传图片的用户中有85%仅仅上传一张图片。

解决方案：团队发现用户不知道使用shift来选择多个图片进行上传，所以加了一个提示，在上传开始前出现上传多张图片的提示。结果数据从85%降低到40%。

纯从这段话中去理解，按以前的习惯我会先列出其中所有的名词：

用户、图片、相册、专辑、第一屏、文件、文件选择功能、上传按钮、java/flash选择、浏览器原声文件选择、浏览器、上传按钮、上传进度、上传量、上传成功率、上传效率、shift选择方式、提示

可以看到短短的一段话中，包含的信息量非常大。（很诧异我们的设计在讲模型的时候几乎没啥，说来说去就是下载、访问、付费）

然后我们来做一些分析，这是一个相册的产品，提供了专辑的管理和图片上传等功能：

首先这里我们可以看到关注的指标上传效率，具体的量化指标是上传量、上传成功率，还可以看到一个隐含指标是单个用户的上传量（团队发现上传图片的用户中有85%仅仅上传一张图片），这是领导关心的（或者应该还是说产品设计团队关心的）；

其次我们可以大致了解这个图片上传的业务流程是怎样的，并且把数值量化进去，以最开始添加图片的用户基数为100来计：

添加图片(100) -》创建专辑，选择专辑类型(87) -》选择文件(57) -》点击上传按钮(52) -》显示上传进度 -》等待进度完成(48)

整体的上传成功率为48%，也就是少于50%的用户能够成功上传图片。这是产品设计人员通常意义讲的数据埋点，在这个流程上面如果说做了数据埋点，记录了所有的操作，但却没有能体现Facebook所做的优化点“将Java/flash facebook文件选择功能改成浏览器原生文件选择功能”。另外即使单纯说上传成功率，这里的上传成功率是指所有用户的上传成功率还是所有用户所有上传记录的上传成功率呢？其实没有一个清晰的定义，这里涉及一个如何把一个用户的一次交互行为通过什么标示完成地连接在一起。我们的数据往往不同的行为由不同的系统或者不同的表来做记录，之间是断裂的，到了最后就演变成按用户能大致关联一下就ok了。

那么我们把以上的名词转化为数据模型应该是怎样的呢？画一个简单的概念模型：

这里没有去细画实体的标识和主要属性，但我们会发现在画概念模型的时候，我们会关注其他的一些信息：比如浏览器、操作系统、网络这些维度；我们会关注添加图片、创建相册、选择文件等这些接触记录是需要统一标识；另外我们可以在选择文件中标注选择方式，在点击上传按钮中增加上传类型。数据的记录不再是零散的，而且数据之间是有关系的。

在这个模型画出来之后，我们可以用模型来进行数据的演练，看上文中所涉及的数据或者指标都应该是如何从模型中得到的。比如上传成功率，我们就应该知道是从进度完成的上传记录除以发起“添加图片”的记录；而将Java/flash facebook文件选择功能改成浏览器原生文件选择功能，我们可以从点击上传按钮的记录中根据上传类型获取到。

这样的数据模型表达方式并不是要去替代原有的PRD中的产品流程，它是数据需求描述的补充，在具体的开发设计中又可以根据此概念模型逐步细化到逻辑模型和物理模型。当然貌似这些现在都还是理想。

最后我还是有一个疑问，这样的图该谁来画呢？产品设计人员、数据分析人员、开发设计人员，抑或是得有个数据架构设计人员的角色？以前给移动做项目的时候没有角色细分，也都做了，但现在貌似大家都是各司其职，而且也都没有这样的习惯，该怎么落下去实在是个难题。不过反正现在无地可落，我还是自己先慢慢琢磨着吧。

教小乐转魔方之“十字坡-刘关张-二龙山-顶上十字坡-小西天-大西天”

先声明，我自己也是才学会怎么把三阶魔方给完整地拼出来，技术也不咋的，还是初级水平。第一次拼出来也是看了Leon提到的文章《普通三阶魔方复原玩法技巧》学会的。不过在自己练习的过程中，也琢磨着怎么吸引小乐来玩，怎么让小家伙会持续转下去直到拼出完整的六面，这还是挺有难度的。你说要理解其中的技巧很有难度，而如果光背公式那在还获得成就和喜悦前就已经被公式给淹没了。

小乐第一次玩魔方的时候，对于拼出一个面还是比较简单的（其他都是无序的），当时只是和她讲了一下怎么让顶(U)上的块来接侧面的，以及如何把底部(D)的块调整到侧面。所以小家伙在拼好一个面时还是觉得挺有意思，经常转一转，完成一个面后展示给我看。

但在想教她往下一步的时候，发现遇到了难题。原本以为很简单的顶上十字，由于要求每个棱块都要颜色对好，小乐怎么都不习惯，就有点厌倦了。（我估计她有点害怕）不过这个时候我也并没有多去和她讲解啥的，只是她偶尔再拼拼一个面玩玩。而在这段时间，结合着魔方小站和今古庸龙魔方网，自己对魔方的认识也有了进一步的理解，尤其是3x3x3的基本解法熟悉之后，这个方法在最后5、6、7的翻角、换角、换边速度上相比前文的还是有优势，且要记的公式也不多，而且以后过渡到CFOP更容易些，决定向小乐推荐这个招法。

不过咋开始呢？动动脑子，还是从她喜欢的小说入手，一步步来，最后我总结的招法名词叫：

十字坡 - 刘关张 - 二龙山 - 顶上十字坡 - 小西天 - 大西天

也就是对应了3x3x3的基本解法中的7个步骤，其中6和7归结为一起算大西天。在这个招法名词中融合水浒、三国和西游记三大名著，也是小乐日常喜欢念叨的人物。

以顶层白色为例：

“十字坡”的招法要解决棱块对齐的问题，改进了一下方法，采用先把白色棱块在黄色底部先拼好，然后逐块和侧面中心块颜色对齐后两圈回，这样小乐就比较容易接受了。同时这个时候小家伙开始学会了观察每个色块都是有它固定的位置的。

“刘关张”就是要完成白色完整的一面。有了十字坡的基础后，小乐开始明白了白色对齐后，边上一周的颜色需要相同，我们就把他们称为“刘关张”三兄弟，而每次要完成一条边，都会有兄弟团聚的过程，经常就念叨关羽和张飞要去接大哥等等之类。这个小乐有一开始的基础就比较快了。

“二龙山”是要完成两层，取自武松上二龙山和鲁智深聚义，口诀是怎么样的呢？武松要和鲁智深聚义，对牢位置，然后：武松一开始先远离二龙山，鲁智深下来接他，武松一看就往回赶，没想把鲁智深给赶跑了，鲁智深再转回来和武松团聚；紧接着把视角转向刘关张（白色），让三兄弟团聚，同时鲁智深和武松也就顺利地汇聚到二龙山了。

到此为止，我们都没有使用任何口诀。

“顶上十字坡”，只有一个口诀“FRUR’U’F’”，也是在这个时候我第一次和她讲口诀，说了这几个字母的含义，碰巧RUF又可以被引申为“路飞”，要成为海贼王的男人，于是口诀很顺地就下来了，接下来只需要识别顶上黄色棱块的形状就ok了。

这个中间的过程还有一点是我和小乐经常性的PK，只要她往前进了一层，我转六面，她完成所学到的层级，一般都是她比我快，这时候小家伙常常会激发和我再往下一步PK的意愿。在到“顶上十字坡”的时候，小家伙的兴趣已经完全被激发了，她已经有意愿想要完成六面了。不过这个时候我倒是故意压慢了下节奏，因为后面几个我自己觉得都是公式，还要判断色块的位置，还是有点麻烦的。

“小西天”完成黄色面，“RUR’URUUR’”，口诀还是挺简单的，继续念路飞，路路路..R，在转的过程中，我们是这样看的，“刘关张”上来，刘备先发起进攻，关羽张飞回营，大哥还不知道，继续走，这时关羽和张飞发现大哥不见了，赶紧回来接，大哥两圈转回来，兄弟团聚一起回营。哈哈，还是刘关张。不过在小西天中麻烦的是可能要多次完成，这里就涉及一个对色块的判断问题，远比“顶上十字坡”要复杂，一开始的时候，我会先告诉她位置，让她先练熟口诀，或者她自己不断转n遍，总能复位。后来开始给她讲判断要诀，就是看“单鱼头”和鱼头朝你时候同时另一角的黄色块要对着你，以及怎么变成“单鱼头”。

“大西天”，取经完成，功德圆满的日子要到了。这里有两步：

先是换角，注意点是黄色面要朝向自己，而不是黄色面朝上，RU’R，“刘关张”上来，然后底层张飞转两圈D2离开，来了个假张飞，（我一开始说刘备，小乐说是张飞，和之前的刘备离开是不同的，说明她自己还是有一套自己的位置排序），R’UR后发现张飞不对，再D2两圈接回真张飞，最后R2两圈刘关张回白色大本营。

接着是换边，也就是棱块互换，RRURUR’U’R’U’R’UR’，路飞一路路路路，注意的地方就是一开始是两个R，在这个时候已经很难抵挡住小乐转出六面的激动了。

这些名词和招法就这么描写，估计只有小乐和我看得明白了。每个小孩子的方法也只有父母自己去体会，什么才是最好的。最后看一段小乐转魔方的视频，记录的成绩是2分57秒。

在Google reader中设置分享到新浪微博

写博客的人貌似越来越少了，尤其是独立博客，原先订阅的feeds很多都长久没有更新了，前两天把feed理了一把，先把80%以上的给砍了，一下子觉得清爽了好多。于是也又有了打开Google reader的兴趣，虽然新改版的界面让我觉得上面的工具栏太宽了些。

看的过程中有时还是难免想随手分享一下，buzz已经over了，twitter和facebook要翻墙，能看到的人也不多，所以想能分享到新浪微博上，可Send to 工具栏上是没有分享到新浪微博的选项。不过打开设置，看了一下Send To是支持自定义的，自己设置一下即可。

Name: 新浪微博

URL: http://v.t.sina.com.cn/share/share.php?title=${title}&url=${url}&source=${source}

Icon Url: http://t.sina.com.cn/favicon.ico

悲剧的self-assigned ip address

从上周五就突然发现mac连公司的wifi连接不上，得到的IP都是169打头的self-assigned ip address，很疑惑。

于是又是关闭wifi，打开wifi，不行；

再renew DHCP，重新获得地址，不行；

看到有的说，要新设置一个location，删掉wifi，再重新添加，也试了，还是不行；

最可笑的重启也试了，不行啊。

可能是公司的无线连接太多了，分配不到IP地址了，这个是比较靠谱的，那就周一赶早来上班，还是连不上。再想办法让IT帮忙释放一些没连接的地址出来，结果还是不行。

实在是晕了，只有直接冲到IT部门去，面对面看了。

第一个问题就是问我的机器名是什么？我拿到机器用到现在根本没去管过它的名字，虽然有的时候打开Terminal，看到那个缺省的有点怪怪的，但也无妨啊。今天一问，才发现机器还没名字，而IT刚上周对这些没名字的都做了限制，不让连。

哎，结论就是没名字的人伤不起啊，机器也是。

最后结果就是修改机器名，再重新连接wifi，搞定！
